# Blog Knowledge Canvas

**「自分のブログの記事構造が見えていない」という課題を起点に、コンテンツ戦略を可視化・加速するナレッジベース。**

WordPressブログの記事をREST APIで取得し、TF-IDFキーワード相関とLouvainクラスタリングで記事間の関係を解析。無限キャンバス上のグラフとして描画し、新記事アイデアを自動提案する。**バックエンド不要・LLM API不要**でブラウザだけで完結する。

---

## なぜ作ったか（背景・課題）

| Before（課題） | After（このツールで解決）|
|---|---|
| 記事が増えるほど「何を書いたか・何が抜けているか」が把握できなくなる | 全記事の相関をグラフで一覧し、空白領域を即座に発見できる |
| ChatGPT等のLLM APIに記事を投げると毎月コストが発生し続ける | ブラウザ完結のTF-IDF解析でAPI費用ゼロ |
| 思いつきで記事を書くためコンテンツの網羅性・戦略性が担保されない | 未結合キーワード対・手薄クラスタから新記事アイデアを自動生成 |
| タグやカテゴリ運用が不十分なブログでも使いたい | タグ0・カテゴリ僅少でもTF-IDF相関が主情報源として機能 |

## 主な機能

**コンテンツ解析**
- WordPress REST APIから全記事を自動取得（プロキシ不要・CORS標準対応）
- HTML・CTA定型文を除去したクリーンなテキストをTF-IDFで解析
- kuromoji.js（IPADIC）による日本語トークナイズ、固有名詞は複合語マージで対応

**グラフ可視化**
- Louvainアルゴリズムでクラスタを自動検出・色分け
- 関連度θスライダーでエッジ密度をリアルタイム調整（発散↔収束）
- ノードクリックで記事のキーワードと関連記事を右パネルに表示
- fcoseレイアウトによる高速・見やすい描画

**新記事アイデア自動生成**
- 未結合キーワード対（連想ギャップ）を検出し提案
- 記事数が少ない手薄クラスタを優先的にサジェスト
- 多様性キャップにより同一テーマ語の重複提案を防止

**データ管理**
- IndexedDBによる自動キャッシュ（差分更新対応）
- JSON書き出し・取り込みで解析結果を保存・チーム間で共有可能

## アーキテクチャ設計の判断

> ここに「なぜその技術を選んだか」を書くことが、採用担当・技術レビューアーへの最大のアピールになる。

**バックエンドレス設計を選んだ理由**
WordPress REST APIはCORSに標準対応しているため、プロキシサーバーを立てる必要がない。静的ホスティング（GitHub Pages等）で完結し、運用コスト・インフラ管理コストをゼロに抑えられる。

**LLM APIを使わない理由**
ブログ記事のコンテンツ戦略支援という用途では、意味的に「近い・遠い」の相対関係が重要であり、絶対的な意味理解は必要ない。TF-IDF＋Louvainの組み合わせで実用的な精度が出ることを実データで検証済み。APIコストが発生しないためユーザーの継続利用コストもゼロ。

**データ層と描画層を分離した理由**
`graphology`（グラフ計算）と`Cytoscape.js`（描画）を独立させることで、将来5,000記事超になった場合は近似最近傍（ANN）にデータ層だけ差し替え、1,000ノード超で描画が重くなった場合はSigma.js（WebGL）に描画層だけ差し替えられる。スケール時の対応コストを最小化する設計。

## 技術スタック

| レイヤー | 採用技術 | 選定理由 |
|---|---|---|
| フレームワーク | Vite + React + TypeScript | 型安全・高速ビルド |

## セットアップ

### 必要環境

- Node.js 18以上
- npm 9以上

### インストール

```bash
git clone https://github.com/<your-username>/blog-knowledge-canvas.git
cd blog-knowledge-canvas
npm install
```

### 開発サーバーの起動

```bash
npm run dev
```

ブラウザで `http://localhost:5173` を開く。

> **初回起動時**：kuromoji辞書（数MB）をCDNから取得する。以降はブラウザキャッシュに保持されるため2回目以降は即時起動。

### プロダクションビルド

```bash
npm run build
npm run preview   # ビルド結果をローカルで確認
```

`dist/` フォルダをそのままGitHub Pages・Netlify・Vercel等にデプロイ可能。

## 使い方

### 1. ブログを解析する

上部の入力欄にWordPressブログのURL（例: `https://haitou-choco-nikki.com/`）を入力し「解析」ボタンをクリック。REST APIから全記事を自動取得・解析する。

### 2. グラフを読む

- 各ノードが1記事を表し、関連度の高い記事がエッジで結ばれる
- 同じクラスタ（テーマ群）の記事は同じ色で表示される
- **関連度θスライダー**を動かすとエッジ密度が変わる（右に絞るほど強い関連だけが残る）

### 3. 記事を深掘りする

ノードをクリックすると右パネルに以下が表示される：

- その記事の特徴キーワード
- 関連度の高い記事一覧

### 4. 新記事アイデアを確認する

右下の**新記事アイデアパネル**に以下が自動表示される：

- **連想ギャップ**：よく登場するがセットで記事になっていないキーワードペア
- **手薄クラスタ**：記事数が少なく掘り下げ余地のあるテーマ群

### 5. 解析結果を保存する

JSONエクスポートで解析結果をファイルに保存できる。次回は同じJSONをインポートして即時再開（再取得・再解析不要）。IndexedDBにも自動保存されるため、同じブラウザなら再訪時に差分更新で再開できる。

## 主要パラメータの調整

`src/config.ts` で以下を変更できる（コード修正はここだけでよい）。

| パラメータ | 説明 | デフォルト |
|---|---|---|
| `initialTheta` | 関連度閾値の初期値 | `0.15` |
| `knnLimit` | 1記事あたりの最大エッジ数 | `10` |
| `topKeywords` | 記事あたりの表示キーワード数 | `10` |
| `ctaMarkers` | CTA定型文除去のトリガーワード | （設定済み）|
| `ideaTermUseCap` | 新記事アイデアでの同一語の最大登場回数 | `2` |

## ユーザー辞書（固有名詞の追加）

「銘柄スカウター」「マネックス証券」のような複合語が分割されて解析精度が落ちる場合は、辞書に追加する。

**手順：**

1. `src/nlp/userdict.ts` の `USER_DICTIONARY` に語を追加する（表記どおりでよい。スペースは自動除去）
2. `src/nlp/tokenize.ts` の `TOKENIZER_VERSION` を1つ増やす

`TOKENIZER_VERSION` を変更すると**IndexedDBキャッシュが自動無効化**され、次回解析時に全記事が再トークン化される。手動でキャッシュを削除する必要はない。

## ディレクトリ構成

```
src/
  ingest/       WP REST APIからの記事取得
  nlp/          clean（HTML/CTA除去）· tokenize（kuromoji）· tfidf
  graph/        build（θ/kNN→graphology）· analyze（Louvain・中心性）
  ideas/        engine（発散/収束提案・多様性キャップ）
  store/        IndexedDBキャッシュ + JSON入出力
  components/   Toolbar · CanvasGraph · SidePanel · IdeaPanel
  pipeline.ts   取得→解析→グラフ→アイデアの統合制御
  config.ts     主要パラメータの一元管理
```
